name: sprint-snapshot
description: Use to render and commit the current sprint state as a reviewable snapshot. Use after any significant status update or at natural sprint milestones (mid-sprint check-in, sprint close).
---

## Goal

Render the current sprintctl sprint state to a committed plaintext snapshot that serves as the shared, diffable record of sprint progress.

## Inputs

- The target sprint ID if the active sprint is not the one to snapshot.
- A loaded project DB environment via `.envrc` or exported `SPRINTCTL_DB`.
- Whether the current workflow expects a standalone snapshot commit or only a regenerated working tree diff.

## Steps

1. Load the project DB first via `.envrc` or exported `SPRINTCTL_DB`.
2. Run `sprintctl render` to produce the current sprint document. If the target is not the active sprint, pass `--sprint-id <id>` if supported by the local workflow.
3. Write the output to `docs/sprint-snapshots/sprint-current.txt`.
4. Review the diff to confirm the snapshot reflects actual sprintctl state rather than manual edits.
5. If the workflow calls for a snapshot commit, stage and commit it separately with `chore: update sprint snapshot`.

## Output contract

- `docs/sprint-snapshots/sprint-current.txt` matches `sprintctl render` output for the intended sprint.
- The user or reviewer can diff sprint-state changes without consulting the local SQLite DB.
- If a commit was made, it is a standalone snapshot commit rather than mixed into unrelated feature work.

## Do not

- Do not edit the snapshot file manually — it is always generated by `sprintctl render`.
- Do not run snapshot updates against an unscoped home-directory DB by accident.
- Do not commit snapshot updates as part of feature commits; keep them as standalone `chore:` commits when committed at all.
